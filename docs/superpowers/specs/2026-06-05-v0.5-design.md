**v0.5 是「开发者自我认知平台」**：

### 2.2 Source 协议（数据源插件化）

**v0.2 问题**：
- `collect_chrome.py` / `collect_edge.py` / `collect_git.py` 各自一套接口
- 加 Firefox 要复制粘贴 80% 代码
- main.py 硬编码 3 个 collect_* 调用

**v0.5 设计**：

```python
# digital_nutrition/sources/base.py
from typing import Protocol
from datetime import datetime
from pathlib import Path
from ..models import Event


class Source(Protocol):
    """数据源协议 - 所有数据源实现统一接口"""
    name: str  # "chrome" / "firefox" / "git" ...

    def is_available(self) -> bool:
        """检查数据源是否可访问（路径存在/配置正确）"""
        ...

    def collect(self, since: datetime | None = None) -> list[Event]:
        """采集事件，过滤 since 之后的"""
        ...


# digital_nutrition/sources/registry.py
class SourceRegistry:
    _sources: list[Source] = []

    @classmethod
    def register(cls, source: Source):
        cls._sources.append(source)

    @classmethod
    def all_available(cls) -> list[Source]:
        return [s for s in cls._sources if s.is_available()]


# Chrome / Firefox / Edge / Git 在各自模块里：
# from .registry import SourceRegistry
# SourceRegistry.register(ChromeSource())
```

**main.py 简化**：
```python
def collect_all_events(since: datetime) -> list[Event]:
    events = []
    for source in SourceRegistry.all_available():
        try:
            events.extend(source.collect(since))
        except Exception as e:
            print(f"Warning: {source.name} failed: {e}")
    return events
```

**Firefox 实现要点**：
- 用 `places.sqlite`（不是 `History`）
- 表名不同：`moz_places` / `moz_historyvisits`
- 列名不同：`url` / `title` / `visit_date` / `visit_count`
- 时间戳：WebKit 微秒（同 Chrome）
- 路径：
  - Windows: `%APPDATA%\Mozilla\Firefox\Profiles\<profile>\places.sqlite`
  - macOS: `~/Library/Application Support/Firefox/Profiles/.../places.sqlite`
  - Linux: `~/.mozilla/firefox/.../places.sqlite`

### 2.3 分类引擎升级（子路径 + 优先级）

**v0.2 问题**：
- 域名级规则粒度太粗：`github.com` 默认 learning，但逛 Issues 是 work
- 用户规则 vs 默认规则合并是 dict 合并，不是优先级
- 无法表达"匹配 A 路径但 B 路径例外"

**v0.5 设计**：

```python
# digital_nutrition/models.py
@dataclass
class Rule:
    pattern: str            # "github.com" 或 "github.com/*/issues"
    category: str           # "work" / "learning" ...
    subcategory: str | None # "code-review" / "reading-list" ...
    priority: int           # 数字越大越优先
    source: str             # "default" / "user" - 仅供调试


# digital_nutrition/classify/engine.py
class Classifier:
    def __init__(self, rules: list[Rule]):
        # 按 priority 降序排
        self._rules = sorted(rules, key=lambda r: -r.priority)

    def classify(self, url: str) -> tuple[str, str | None]:
        """返回 (category, subcategory)"""
        for rule in self._rules:
            if self._matches(rule.pattern, url):
                return rule.category, rule.subcategory
        return "other", None

    @staticmethod
    def _matches(pattern: str, url: str) -> bool:
        # glob 风格: "github.com/*/issues" 匹配 "github.com/user/repo/issues"
        import fnmatch
        return fnmatch.fnmatch(url, f"*{pattern}*")
```

**用户规则 CLI**：
```bash
# 添加规则
digital-nutrition rules add "github.com/*/issues" --category work --priority 100

# 列出规则
digital-nutrition rules list

# 删除规则
digital-nutrition rules remove "github.com/*/issues"

# 规则存储在 ~/.config/digital-nutrition/user_rules.json
```

**默认规则扩展**（v0.5 新增子路径）：
```json
{
  "work": [
    {"pattern": "github.com/*/issues", "subcategory": "issue-tracking"},
    {"pattern": "github.com/*/pull", "subcategory": "code-review"},
    {"pattern": "github.com/*/commits", "subcategory": "code-review"}
  ],
  "learning": [

  ]
}
```

### 2.4 多维度人格系统

**v0.2 问题**：
- 7 种人格互斥，只能选 1 个
- 实际用户往往是混合的：50% code + 30% learning + 20% entertainment
- "多元探索者"是兜底，太粗

**v0.5 设计**：

```python
# digital_nutrition/analysis/persona.py
@dataclass
class PersonaScore:
    """多维度人格评分（0-1 各维度）"""
    code: float = 0.0
    learning: float = 0.0
    work: float = 0.0
    entertainment: float = 0.0
    news: float = 0.0
    social: float = 0.0
    shopping: float = 0.0
    other: float = 0.0

    @property
    def primary(self) -> str:
        """主标签 - 最高分维度"""
        dims = self.as_dict()
        top_cat = max(dims, key=dims.get)
        return PRIMARY_PERSONAS.get(top_cat, "🌐 多元探索者")

    @property
    def tags(self) -> list[str]:
        """多标签 - 所有 >15% 的维度"""
        return [cat for cat, score in self.as_dict().items() if score >= 0.15]

    @property
    def diversity(self) -> float:
        """香农熵 - 时间分配多样性"""
        import math
        scores = [s for s in self.as_dict().values() if s > 0]
        return -sum(s * math.log(s) for s in scores)

    def as_dict(self) -> dict[str, float]:
        return {
            "code": self.code, "learning": self.learning, ...
        }
```

**主标签**（保持 v0.2 的 7 种人格作为"主标签"）：
- 🧱 代码机器人 (code > 50%)
- 📚 学习永动机 (learning > 40%)
- 🎬 娱乐至上 (entertainment > 30%)
- 📰 资讯焦虑 (news > 40%)
- 🔍 观察者 (social > 30%)
- ⚖️ 平衡大师 (code/learning/work 各 15-30%)
- 🌐 多元探索者 (兜底)

**多标签**（v0.5 新增）：
- 同时显示 Top 3 维度的标签
- 例：🧱 + 📚 + ⚖️ 标签同时出现

**显示**（HTML 报告）：

### 2.5 活跃度检测

**v0.2 问题**：
- Chrome 记录 visit duration = 实际挂机时长 + 真实浏览
- 报告里硬编码提示"估算"—— 治标不治本

**v0.5 设计**：
- **短间隔检测**：如果两次 visit 间隔 < 30s 且都同域名 → 算挂机，duration 减半
- **超长截断**：单次 duration > 30min → 截断到 30min
- **加可调参数**：用户在 `~/.config/digital-nutrition/config.toml` 可调

```toml
# ~/.config/digital-nutrition/config.toml
[activity]
idle_threshold_seconds = 30
max_session_minutes = 30
deduplicate = true
```

## 3. 新功能清单

### 3.1 Firefox 支持（v0.3 候选 #1）

| 维度 | Chrome | Firefox |
|------|--------|---------|
| 数据库 | `History` | `places.sqlite` |
| URL 表 | `urls` | `moz_places` |
| 访问表 | `visits` | `moz_historyvisits` |
| 列名 | `url` / `title` / `visit_time` / `visit_duration` | `id` / `url` / `title` / `visit_date` |
| 路径 | `User Data/<profile>/History` | `Profiles/<profile>/places.sqlite` |

**注意**：
- Firefox 没有 `visit_duration` 字段！需用 `visit_date` 间隔推算
- 简化：第一次 visit → 0 持续；后续 visit → 与上次 visit 的时间差
- 损失精度但 Firefox 用户本来就不多

### 3.2 分享卡 PNG（v0.3 候选 #2）

**3 种实现方案**（选最简）：

**方案 A：Pillow 手绘**（推荐）
- 优势：无外部依赖 / 离线 / 速度快
- 劣势：图表需简化（无 ECharts）

**方案 B：ECharts server-side 渲染**
- 优势：图表精美

# v0.5 v1 反思（self-review）

> 这一节我自己挑刺，找设计的薄弱处。

## 反思 1：Phase 1 顶层重构风险极高

**问题**：
- 把 11 个文件全改成 `from digital_nutrition.X` 是大量机械修改
- 任何一处漏改 → 整个 package 不能用
- 测试要全部重写导入路径
- v0.2 已经稳定工作，破坏它成本高

**可能的失败**：
- 改完发现某个子模块没在 `__init__.py` 导出 → 跑不起来
- `pyproject.toml` 配置错 → `pip install -e .` 失败
- v0.2 引入的 `scripts/__init__.py` 改造没考虑到 E2E 测试用 `subprocess` 跑

**缓解**：
- Phase 1 严格按"先建新结构 → 复制代码 → 改导入 → 跑测试 → 删旧结构"5 步走
- 旧 `scripts/` 暂不删，直到 `from digital_nutrition.X` 全跑通

## 反思 2：Source 协议过度设计？

**问题**：
- v0.2 只有 3 个 source，加 1 个 Firefox，用 Protocol 真的需要？
- `SourceRegistry` 是 singleton 全局状态，难测试
- Protocol vs ABC vs 鸭子类型——选哪个？

**评估**：
- 如果用 Protocol（PEP 544）→ 静态类型支持，运行时零开销
- 如果用 ABC → 强制 `is_available()` 实现，但更重
- 如果用鸭子类型 → 最简单，但 IDE 提示差

**建议**：用 Protocol + 简单 `list[Source]` 替代 Registry（避免全局状态）

## 反思 3：多维度人格反而降低可读性？

**问题**：
- 7 种人格用户能记住，多维度评分用户看不懂
- "🧱 + 📚 + ⚖️" 三个标签放一起是啥意思？
- 香农熵对普通用户没意义

**可能的失败**：
- 报告变得"数据丰富但故事弱"
- 用户报告发出去别人看不懂

**缓解**：
- 主标签保留 v0.2 的 7 种人格（v0.2 用户已熟悉）
- 多维度只作为"维度条"展示（不强行组合）
- 香农熵只显示为"分散度 0.8"这样的数字，不解释

## 反思 4：规则 CLI 是 YAGNI？

**问题**：
- v0.2 的 `user_rules.json` 直接编辑 JSON 已经能覆盖 80% 用例
- 加一个 subcommand 写 JSON 是过度工程
- 真正痛点是"我不知道该加什么规则"——这是规则推荐问题，不是 CLI

**评估**：
- 用 CLI 命令：10 分钟能加一条规则
- 用编辑器改 JSON：也是 10 分钟
- 提升有限

**建议**：先不做 CLI，只在 SKILL.md 加"如何手动加规则"小节。如果用户多次要求再加 CLI。

## 反思 5：分享卡 PNG 是 YAGNI？

**问题**：
- v0.2 的 HTML 报告已经能在浏览器里看，截图也能分享
- 加 PNG 需要：
  - Pillow 依赖
  - 字体文件（5MB+）
  - 绘制代码（200 行+）
- ROI 存疑

**评估**：
- 如果用户不主动分享 → 写了也白写
- 如果用户想分享 → 截图工具 5 秒搞定

**建议**：先做，但用最简实现（纯文字 + 柱状条，无字体文件依赖）

## 反思 6：Phase 计划是不是太乐观？

**问题**：
- 4 phases / 4 commits / 8-12 小时
- 但 v0.1 实际花了 4 小时做 11 模块，v0.2 花了 2 小时做 2 模块
- v0.5 的工作量 ≈ v0.1 + v0.2 翻倍
- 实际可能要 16+ 小时

**缓解**：
- 每个 phase 限制在 2-3 小时内
- 超过就降级（不做分享卡、不做规则 CLI）

## 反思 7：src-layout 真的是必要的吗？

**问题**：
- v0.2 用 `flat-layout`（package 在根目录）也能 work
- src-layout 主要解决"测试时 import 本地代码"vs"装好后 import"不一致
- 但 v0.2 是单 package，没子包，问题不严重

**评估**：
- v0.5 子目录多（sources/ / classify/ / analysis/ ...）
- src-layout 让 `from digital_nutrition.X import Y` 显式表达 package 边界
- 但 flat-layout 也能做到

**建议**：保持 flat-layout（避免 v0.1 → v0.2 → v0.5 三次重命名）—— **这条需要再讨论**

## 反思 8：Firefox 没 visit_duration 怎么办？

**问题**：
- v0.2 假设所有 source 都有 duration
- Firefox 缺这个字段
- 用 `visit_date` 间隔推算 → 第一次 visit 是 0，最后一次是窗口边界

**可能的失败**：
- Firefox 数据全是 0 持续 → persona 永远是"数据不足"
- 报告里 Firefox 占比 0%，看起来没贡献

**缓解**：
- Firefox 用"下次 visit 时间差"算法：
  - 对每个 URL 的连续 visits，计算相邻 visit 的时间差
  - 取差值作为 duration（设上限 30min）
- 这和 Chrome 的 `visit_duration` 是不同语义，要在报告里说明

# v0.5 v2（反思后优化版）

> 基于 v1 反思，调整如下：

## 优化 1：保留 flat-layout，去掉 src/

**理由**：
- flat-layout 已经 work，重构风险大
- v0.5 子目录多也无所谓，Python package 不限制子目录深度

## 优化 2：Source 用 Protocol + 简单 list，不用 Registry

**理由**：
- Protocol 类型提示足够
- 全局 Registry 难测试 + 增加耦合
- 简单 `list[Source]` 在 main.py 里显式管理

```python
# main.py
def collect_all_events(since):
    events = []
    for source in DEFAULT_SOURCES:  # 简单 list
        if not source.is_available():
            continue
        try:
            events.extend(source.collect(since))
        except Exception as e:
            print(f"Warning: {source.name} failed: {e}")
    return events


# digital_nutrition/sources/__init__.py
from .chrome import ChromeSource
from .edge import EdgeSource
from .firefox import FirefoxSource
from .git import GitSource

DEFAULT_SOURCES = [ChromeSource(), EdgeSource(), FirefoxSource(), GitSource()]
```

## 优化 3：多维度人格保持"主标签 + 维度条"，不做多标签组合

**理由**：
- 多标签组合可读性差
- 主标签（v0.2 的 7 种）+ 维度条（7 维 0-1 评分）= 信息密度合适

**显示**：
```

学习     █████         0.28

娱乐     █             0.05
```

## 优化 4：规则 CLI 推迟到 v0.6，先做最小化的"已添加规则"提示

**理由**：
- CLI 是 YAGNI
- 但"用户加了什么规则"应该有地方看到

**调整**：
- v0.5 不加 `rules` subcommand
- 加 `digital-nutrition report` 的 metadata 块显示"用户自定义规则：3 条"
- CLI 等 v0.6 用户多次要求再做

## 优化 5：分享卡用最简实现（无 Pillow 依赖）

**理由**：
- 避免新依赖
- 用 SVG + cairosvg（10KB）or 直接输出文本卡

**调整**：
- v0.5 分享卡 = **纯文本卡片**（`.txt` 文件），含：
  - 人格主标签
  - Top 3 类别百分比
  - 2-3 条洞察
  - 生成时间
- PNG 留给 v0.6，需要时再加 Pillow 依赖
- 或者用 ECharts 的 `getDataURL()` 在浏览器里生成 base64 PNG，Python 端解码保存

**新方案**：浏览器端生成 PNG
```javascript
// 在 HTML 报告里加按钮："保存分享卡"
button.onclick = () => {
    const canvas = document.createElement('canvas');
    canvas.width = 1200; canvas.height = 630;
    const ctx = canvas.getContext('2d');
    // ... 绘制
    const png = canvas.toDataURL('image/png');
    // 触发下载
    const link = document.createElement('a');
    link.download = 'nutrition-card.png';
    link.href = png;
    link.click();
};
```
**优势**：零 Python 依赖，零字体问题，所见即所得
**劣势**：需要用户手动点按钮（不能 CLI 一键生成）

**最终选择**：
- v0.5 做**纯文本分享卡**（自动生成，无需用户操作）
- v0.5 HTML 加**手动保存 PNG 按钮**（不强制，要用就点）
- v0.6 再考虑 Pillow 全自动方案

## 优化 6：Phase 计划改为"3 阶段渐进交付"

**理由**：
- 4 phases 太多，节奏太散
- v0.2 3 phases 节奏好，v0.5 保持

**新计划**：

| Phase | 内容 | 风险 | 时间 | commit |
|-------|------|------|------|--------|

**总预计 7 小时**（vs v0.1 4h + v0.2 2h = 6h）。**注意 Firefox 暂时不做**（v0.6 再做，避免 v0.5 范围爆炸）。
